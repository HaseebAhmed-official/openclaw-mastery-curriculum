# Weak Submission Red Flags

## Purpose

This file helps instructors and students recognize low-quality work before grading begins.

## Red flags

- The submission says "it works" but provides no evidence path.
- The artifact uses security language but never names the actual trust boundary.
- The student describes sandboxing as if it eliminates host risk completely.
- The design exposes the gateway publicly without explaining why that exposure is justified.
- The submission talks about automation but never addresses auditability or durable authority.
- The student assumes provider or model defaults without recording version or release assumptions.
- The design review contains architecture words but no rejected alternatives.
- The capstone summary overclaims "enterprise-ready" without operational runbooks, rollback thinking, or threat analysis.
- The student confuses multi-agent routing with secure isolation between hostile users.
- The artifact uses plugins, hooks, or external code with no supply-chain note.

## Usage

If a submission shows multiple red flags from this list, it should receive deeper review before any strong grade is awarded.
