# Track Evaluation Sheets

## Purpose

These sheets give graders a fast checklist aligned with each track's non-negotiable outputs.

## Operator track

- workflow deployment documented
- channel policy included
- troubleshooting runbook included
- secure baseline note included
- prompt injection exposure considered
- operator recovery path explained

## Production / DevOps track

- deployment architecture included
- ingress and identity flow explained
- update and rollback note included
- webhook or detached-work control note included
- operational runbook included
- release-aware assumptions documented

## Security / Hardening track

- threat model included
- audit review included
- prioritized remediation plan included
- residual-risk statement included
- webhook, hook, plugin, and external-content risks considered
- sandboxing and formal verification limits correctly stated

## Plugin Developer track

- manifest or extension artifact included
- schema or config model included
- install/update behavior documented
- compatibility assumptions stated
- validation plan included
- precedence or ownership issues considered

## Contributor / Core track

- contribution proposal included
- architecture impact note included
- validation plan included
- docs plan included
- scoped contributor guidance considered
- changed-scope validation awareness shown

## Local Models track

- architecture included
- hardware assumptions included
- quality and safety tradeoff note included
- fallback and cost note included
- hosted-vs-local justification included
- strong-model requirements acknowledged for tool-enabled work
